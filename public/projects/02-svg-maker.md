# svg maker
_teaching a machine to draw_

I am building (it is ongoing) a tool that generates SVG icons from short, natural language prompts — matching existing library styles, shape conventions, and grid constraints

i've created about 1,000 finalized svgs for Block libraries. they have consistent stroke weights, standardized corner radii, precise visual centers. my lofty goal is to generate new icons that belong next to those, but mostly I like the process of tweaking and learning about LLMs.

### learning the language

I started with the fundamentals. SVGs are instructions, not pixels. each icon lives on a 24x24 unit grid:

`<svg viewBox="0 0 24 24" fill="none" xmlns="http://www.w3.org/2000/svg">`

paths are built from a handful of commands. `M` moves the pen. `L` draws a line. `Z` closes the path.

### teaching the machine



it is suprisingly difficult to generate usable, unique SVGs, given how incredible these models are at creating images. but generating a PNG is more like art: the model imagines pixels. generating an SVG is math: it has to write precise coordinates, calculate curves, and produce code that actually parses.

as a baseline, I started by only adding basic instructions for 24x24 SVG icon generation:
+ stroke-width: 2, stroke-linecap: butt, stroke-linejoin: round. stay on the 24x24 grid
+ 1px padding, unless the path is a bezier
+ round coordinates to the nearest .5. use cubic béziers for organic curves, arcs for perfect circles.

early outputs are valid SVG but..... soooo bad (and clearly not inheriting any style nuance). they had inconsistent strokes, rounded end caps, points were 1 pixel outside the grid, a lot of overlapping paths, would scare children, etc.

then I added svg best practices and basic styling rules:
+ H/V commands for straight lines, S/T for smooth curves, consistent arc radii, symmetrical control points
+ Path joining rule — join paths ending on same coordinates into a single path
+ Curve reuse rule — reuse curve patterns from reference examples for visual consistency
+ Quoted prompt rule — prompts in "quotes" invite abstract/conceptual interpretation
+ Structural simplicity rules — one path per continuous shape, prefer simplest stroke, minimize decorative detail, one iconic identifier for broad subjects
+ Enforce strokeLinecap="butt" — strengthened in both system and user prompts
+ No-overlap rule — detail paths must not overlap each other or parent strokes
+ No-eyes rule — unless user prompt specifically asks for eyes

I learned about text + image embeddings when I was tasked with improving search and retrieval in the assets cabinet. so i added embeddings here:
→ when a prompt comes in, the system searches both by description _and_ by visual similarity
→ results merge by keeping the highest score per icon
→ the top 3 matches get fed into the prompt as reference SVGs, so the model can riff on proven geometry.

at a certain point, I could tell the model was making decisions on metphor, but couldn't always figure out the intent. so i had it label each path semantically.

labeling paths helps a lot. every `<path>` gets a comment: `<!-- handle -->`, `<!-- blade -->`, `<!-- arrow-head -->`. once the model understands the anatomy of an icon it reasons about structure instead of just coordinates.

### evals

to measure progress and test various models, I added an eval pipeline. each run generates a batch of icons from the same set of prompts, scored on validity (does it parse), grid alignment, stroke consistency, path efficiency, and visual similarity to the reference. it makes prompt changes measurable with actual data. why isn't everyone talking about evals??????


### things I've learned other than the importance of evals

**examples > instructions.** I can write paragraphs explaining stroke conventions, but three well-chosen reference SVGs in the prompt teach more than any of it. the model learns style by seeing it, not by being told about it.

**the viewBox is sacred.** anything outside it gets clipped. anything too small inside it looks lost. key shapes should fill 80-90% of the grid.

**naming creates understanding.** labeling paths — `<!-- left-wing -->`, `<!-- envelope-flap -->` — makes edits smarter. the model can "find the handle and make it longer" instead of guessing which segment to modify.

### what's next

the system prompt has gotten me pretty far, but there's a ceiling. the model is interpreting instructions about drawing, not learning from the drawings themselves.

the next step is supervised fine-tuning. take the thousands of hand-drawn SVGs from existing libraries, pair each with a natural language description, and train the model on that mapping directly. not "here are rules about how to draw" but "here are ten thousand examples of what good looks like."

SFT won't replace the system prompt. it'll replace the need for the prompt to carry so much weight. the prompt focuses on _what_ to draw, the model already knows _how._
