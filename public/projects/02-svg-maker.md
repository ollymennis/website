# svg maker
_teaching a machine to draw_

I am building (it is ongoing) a tool that generates SVG icons from short, natural language prompts — matching existing library styles, shape conventions, and grid constraints

### the (lofty) goal
I've created about 1,000 finalized svgs for Block libraries. they have consistent stroke weights, standardized corner radii, precise visual centers. my goal is to generate new icons that belong next to those, but mostly I enjoy the process of tweaking, testing, and learning about LLMs.

### teaching the machine

given how incredible these models are at creating images, it is suprisingly difficult to generate usable, unique SVGs. but generating a PNG is more like art: the model imagines pixels. generating an SVG is math: the model has to write precise coordinates, calculate curves, and produce code that actually parses.

as a baseline, I started by only adding foundational instructions for 24x24 SVG icon generation:
+ stroke-width: 2, stroke-linecap: butt, stroke-linejoin: round. stay on the 24x24 grid
+ 1pt padding, unless the path is a bezier
+ round coordinates to the nearest .5. use cubic béziers for organic curves, arcs for perfect circles

early outputs are valid SVG but..... soooo bad (and clearly not inheriting any style nuance). they had inconsistent strokes, rounded end caps, grid issues, a lot of overlapping paths, would scare children, etc.

### evals

to measure progress and test various models, I added an eval pipeline.

each run generates a batch of icons from the same set of prompts, scored on validity (does it parse), grid alignment, stroke consistency, path efficiency, and visual similarity to the reference. it makes prompt changes measurable with actual data. why isn't everyone talking about evals??????


### things I've learned other than the importance of evals

**show the model, don't tell.** SVG generation is closer to code generation - a few well-crafted reference SVGs paired with a concise system prompt consistently outperforms prescriptive instructions

**keep prompts separate.** separating prompts from source code makes it way easier to make deliberate changes without having to worry about minor LLM mistakes and confusion
